## Telegram “lag” investigation (OpenClaw)
- Observed: Telegram message enqueue → run start is fast (~100–200ms), so not likely Telegram polling/webhook delay.
- Main latency is LLM run duration (examples in logs: ~6–7s and ~18s) + occasional tool calls.
- Gateway process `openclaw-gateway` was using ~22% CPU when checked; Foundry proxy running too.
- Telegram channel config: `streamMode: "partial"` (possible UX cause: responses arrive in bigger chunks).
- Suggested fixes: enable more aggressive streaming (if supported), stagger cron jobs to reduce contention, restart gateway when sticky, route quick replies to faster model.

## Azure VM resize idea (East US 2)
- Current VM metadata: `Standard_E2s_v3` in `eastus2` (2 vCPU / ~16 GiB RAM).
- Goal: more CPU, same memory → recommend moving to 4 vCPU / 16 GiB:
  - `Standard_D4s_v5` (Intel) or `Standard_D4as_v5` (AMD, often cheaper).
- Rough on-demand cost deltas (compute-only, 730h/mo; estimates from public calculators):
  - E2s_v3 ~$0.126/hr (~$92/mo)
  - D4s_v5 ~$0.192/hr (~$140/mo) → +~$48/mo
  - D4as_v5 ~$0.172/hr (~$126/mo) → +~$34/mo
- Resize requires stop/deallocate; caution about public IP changes if not static.
