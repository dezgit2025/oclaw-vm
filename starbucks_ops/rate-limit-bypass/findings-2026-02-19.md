# Starbucks WiFi Rate-Limit Bypass Testing

**Date:** 2026-02-19
**Location:** Starbucks (public WiFi)
**Mac IP:** 172.16.225.184 (en0)
**VM:** oclaw2026linux (Standard_E2s_v3, East US 2)

---

## Baseline Measurements

| Test | From | To | Speed |
|------|------|----|-------|
| Cloudflare 10MB | VM → Cloudflare | direct (no Mac) | **~408 Mbps** |
| OVH 10MB | VM → OVH (Europe) | direct | ~36 Mbps |
| iperf3 1 stream | Mac → VM (SSH tunnel) | port 22 | **~15 Mbps** |

**Conclusion:** VM has plenty of bandwidth. Bottleneck is Mac → Azure path through Starbucks WiFi.

---

## Methods Tested

### 1. SSH on Alternative Ports (TCP)
Tested if Starbucks shapes specific ports differently.

| Port | Protocol | Recv Mbps | Burst Peak |
|------|----------|-----------|------------|
| 22 (SSH) | TCP | 14.1 | 50 |
| 443 (HTTPS) | TCP | ~similar | — |
| 993 (IMAP) | TCP | 15.8 | 69 |
| 123 (NTP) | TCP | 14.9 | 94 |

**Result:** All ports settle to ~15 Mbps. No per-port shaping.

### 2. WireGuard VPN (UDP 443)
Tested if UDP on port 443 (looks like QUIC) bypasses shaping.

| Method | Recv Mbps | Burst Peak |
|--------|-----------|------------|
| WireGuard UDP 443 | 13.9 | 47 |

**Result:** Slightly worse than TCP. ISP/Starbucks shapes UDP more aggressively.

### 3. ByeDPI TCP Packet Manipulation
Used ByeDPI (TLS ClientHello splitting, disordered packets, fake TTL packets) to confuse DPI middleboxes.

| Method | Recv Mbps | Burst Peak |
|--------|-----------|------------|
| ByeDPI + SSH | 14.5 | 62 |

**Result:** No improvement. Shaping is not DPI-based (not inspecting protocols).

### 4. Parallel Stream Aggregation
Tested if multiple simultaneous connections get more total bandwidth (per-flow vs per-device shaping).

| Streams | Total Recv Mbps |
|---------|----------------|
| 1 | 15.1 |
| 4 | 6.7 |
| 8 | 6.3 |

**Result:** More streams = LESS bandwidth. Shaping is per-device, not per-flow. Multiple Docker proxy containers would not help.

### 5. DNS Tunneling (Not Tested)
Would require a domain with NS records pointing to VM. Typical speed is 50-500 Kbps — much slower than current 15 Mbps. Not worth pursuing.

---

## Key Findings

1. **Shaping is per-device (MAC address)** — not per-port, per-protocol, or per-flow
2. **~15 Mbps is the hard cap** per device on Starbucks WiFi to Azure
3. **VM itself is fast** (~408 Mbps to Cloudflare) — bottleneck is entirely the Starbucks WiFi link
4. **UDP is shaped more aggressively** than TCP
5. **DPI bypass doesn't help** — shaper is not protocol-aware, just bandwidth-capping per MAC

---

## Approach That Could Work: Multi-Device Bonding

Since shaping is per-device, adding a second device with a different MAC gets a second ~15 Mbps allotment.

### Option A: Android Phone WiFi Tethering (FREE — test tomorrow)
- Android connects to Starbucks WiFi (separate MAC = separate 15 Mbps session)
- USB tether to Mac (Android passes WiFi through USB, unlike iOS)
- Mac bonds en0 (WiFi) + USB Ethernet (Android tether)
- Expected: ~30 Mbps aggregate

**Important:** iOS cannot do this — iPhone USB tether only shares cellular, not WiFi.

### Option B: GL.iNet Travel Router (~$30-70)
- Travel router connects to Starbucks WiFi with spoofed MAC
- Provides Ethernet out → USB-C Ethernet adapter → Mac
- Bond WiFi + Ethernet on Mac
- Can add multiple travel routers for more sessions

### Option C: USB WiFi Adapter (NOT viable on Apple Silicon)
- macOS on Apple Silicon has no kernel extension support
- No USB WiFi adapters work natively
- Dead end

### Bonding Tool
```bash
# dispatch-proxy: round-robin SOCKS proxy across multiple interfaces
npm install -g dispatch-proxy
dispatch start en0 en8   # en0=WiFi, en8=USB Ethernet from tether/router
# Creates SOCKS proxy on localhost:1080 that alternates between both links
```
