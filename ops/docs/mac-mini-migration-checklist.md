# Mac mini Migration Checklist (OpenClaw)

_Date: 2026-02-21_

This checklist moves the OpenClaw deployment from the current **Linux VM** to a **physical Mac mini** while preserving:
- OpenClaw config + skills
- Ops scripts/docs
- Memory notes
- Important non‑Git state (tokens, selected logs/JSONLs)

It also documents what is **not** backed up by Git and how to move it safely.

---

## 0) Goals + Migration Strategy

### Recommended strategy (safe + repeatable)
1) **Git** remains the canonical backup for code/docs/skills.
2) **Encrypted backup** (existing workflow) remains the canonical backup for non‑Git artifacts (logs, jsonl, tokens if you choose, etc.).
   - We will **reuse the same AES‑256‑GCM encryption key** (stored in 1Password) on the Mac mini.
   - Prefer restoring from the encrypted backup archive whenever possible.
3) Transfer the encrypted bundle via **Tailscale tunnel** (preferred) or **Dropbox staging** (optional).

### Simple flow (ASCII)

```
Linux VM (current)                         Mac mini (target)
------------------                         -----------------

  [Git repo]  --------------------------->  git clone -> ~/.openclaw/workspace
      |                                                (skills/, ops/, docs/, memory/)
      |
      |  [Encrypted backup bundle]
      |        (AES-256-GCM, key in 1Password)
      |
      +--(preferred) Tailscale tunnel/scp -->  restore bundle -> tokens/logs/jsonl (as needed)
      |
      +--(optional) Dropbox staging ------->  download bundle -> restore -> delete staging

Then:
  copy/adjust ~/.openclaw/openclaw.json -> start gateway -> run acceptance tests
```

### Key principle
Anything in `.gitignore` is **not** guaranteed to migrate unless we explicitly include it in one of:
- encrypted backup archive, or
- Dropbox staging bundle, or
- re-auth/recreate on the Mac.

## 1) What Git Already Covers (clone on the Mac)

Git backup includes (source of truth):
- `skills/`
- `ops/scripts/`, `ops/docs/`
- `docs/`
- `config/`
- `memory/`
- workspace root docs (`GLOBAL_FOLDER_STRUCTURE.md`, etc.)

Reference: `ops/docs/BACKED_UP_IN_GIT.md`

### On the Mac
- Install OpenClaw
- `git clone` the repo into `~/.openclaw/workspace`

## 2) What Git Does **NOT** Cover (must migrate separately)

These are intentionally excluded by `.gitignore` and/or policy:

### A) Secrets & tokens
Commonly excluded patterns:
- `credentials.json`
- `token.json`
- `*.token.json`
- `.env.*`
- `*.key`

Reference: `ops/docs/EXCLUDED_DB_FILES.md` and `.gitignore`

### B) Logs + JSONLs + runtime artifacts
- `logs/` (entire folder)
- `*.jsonl`
- `*.log`, `*.cache`

### C) Databases
- `*.db`, `*.sqlite`

### D) Cookies
- YouTube cookies are explicitly sensitive and should **never** go to Git.

## 3) Inventory: “Don’t Miss Anything” Checklist (Linux VM)

Run this inventory before migration and save the output into the Dropbox staging bundle (as text files):

### Workspace + OpenClaw config
- `~/.openclaw/openclaw.json`
- `~/.openclaw/agents/` (sessions are usually large; optional, but you can stage recent ones)
- `~/.openclaw/workspace/` (already mostly covered by Git)

### OAuth tokens / API tokens (likely locations)
> Exact filenames can vary by skill.

- Google Drive token(s):
  - `~/.config/openclaw-gdrive/token-openclawshared.json`
- Gmail token(s):
  - `~/.config/openclaw-gmail/` (e.g., `token-alerts.json`, `token.json` depending on skill)
- Any other provider tokens:
  - `~/.config/openclaw-*/`

### Tailscale
- Usually easiest to re-auth on the Mac.
- Still capture current state as reference:
  - `tailscale status`
  - `tailscale ip -4`
  - exit node + routes configuration (document only)

### Python venvs
- Don’t copy venv folders across OSes.
- Instead capture:
  - `pip freeze` outputs (per venv)

### Cron / scheduled jobs
We can rebuild cron jobs from OpenClaw’s internal cron list, but capture it anyway:
- `openclaw cron list` (or use the cron tool list output)

## 4) Dropbox Migration Staging Folder (optional)

### Preferred transfer mechanism
Use the **existing encrypted backup method** (AES‑256‑GCM) and reuse the same encryption key stored in 1Password.

Dropbox should be used only as a temporary staging point *if needed* (e.g., for fast LAN-less transfer).

### Recommended approach (if using Dropbox)
Create a temporary folder in Dropbox, e.g.:
- `Dropbox/OpenClaw-Migration-Staging/<date>/`

Upload **only the encrypted backup artifact(s)** (not raw tokens), restore on the Mac mini, then delete the staging folder.

### Why
- Dropbox is convenient but increases exposure for secrets.
- Using the encrypted archive means Dropbox never sees token contents in plaintext.

### What goes into the staging bundle (if using Dropbox)
Ideally: **only encrypted artifacts** generated by the existing encrypted backup workflow.

If you *must* stage plaintext files (not recommended), this would be the structure — but prefer putting these inside the encrypted archive instead:

```
OpenClaw-Migration-Staging/
  README.txt
  openclaw.json
  tokens/
    openclaw-gdrive-token-openclawshared.json
    openclaw-gmail-token-alerts.json
    ...
  inventory/
    tailscale_status.txt
    open_ports.txt
    cron_jobs.txt
    node_version.txt
    python_version.txt
    kernel_version.txt
    pip_freeze_venv-gmail.txt
  logs-selected/
    (only if you want to preserve specific jsonl)
```

### Token guidance
- Best: **re-auth on the Mac** OR restore tokens from the encrypted archive.
- If restoring tokens, only include what you need on day 1 (Drive token, Gmail-alerts token).

## 5) Migration Execution Steps

### Step 1 — Prepare Mac mini (fresh install)
- Install Node.js (LTS) + Python 3
- Install OpenClaw
- Install Tailscale

### Step 2 — Restore workspace from Git
- Clone repo into `~/.openclaw/workspace`
- Verify folder structure matches `GLOBAL_FOLDER_STRUCTURE.md`

### Step 3 — Restore OpenClaw config
- Copy `openclaw.json` to `~/.openclaw/openclaw.json`
- Adjust any Linux‑specific paths:
  - `/home/desazure/...` → `/Users/<you>/...`

### Step 4 — Restore tokens (from Dropbox staging)
- Place token files into:
  - `~/.config/openclaw-gdrive/`
  - `~/.config/openclaw-gmail/`
- Ensure permissions:
  - `chmod 600` on token files

### Step 5 — Recreate venv(s)
- Create `.venv-gmail` on macOS
- Install required packages based on captured `pip freeze`/requirements

### Step 6 — Start OpenClaw gateway + verify
- Start gateway locally
- Confirm it can:
  - respond in Telegram
  - run a small tool/script
  - access Drive using token

### Step 7 — Rebuild scheduled jobs
- Recreate cron jobs via OpenClaw config/tools
- Verify schedules/timezones

### Step 8 — Delete staging bundle
- After verification, delete the Dropbox staging folder.

## 6) “Things That Commonly Get Missed” (extra safety)

- **Hidden config dirs:** `~/.openclaw/`, `~/.config/openclaw-*`
- **Service files:** systemd units don’t apply on macOS (you’ll use launchd)
- **Webhook URLs / API keys** in env files
- **Token scopes differ per machine:** sometimes easiest to re-auth to avoid weird failures
- **Path assumptions** inside scripts (hard-coded `/home/desazure/...`)
- **Binary dependencies** (yt-dlp, ffmpeg, etc.)

## Appendix B — Hardening note (before/after migration)
The weekly hardening audit currently flags several 🔴 items (gateway bind, firewall posture, token permissions, git hygiene). Migration to the Mac is a good time to fix these by default:
- bind gateway to localhost / tailscale-only
- enable macOS firewall + restrict listeners
- tighten token file perms
- ensure sensitive artifacts never enter Git
