### Round 2 — RRF Fusion (COMPLETE)

**Goal:** Replace dedup+priority sort with Reciprocal Rank Fusion in `cmd_recall()`.

**Production code changes (`mem-source-code/smart_extractor.py`):**

1. **Added `rrf_fuse()` function** (line ~2019):
   - Takes list of per-query result lists
   - Computes `RRF_score(mem) = SUM[1/(k + rank + 1)]` across all sub-query lists
   - Sorts by RRF score descending, tag priority as tiebreaker
   - Memories appearing in multiple sub-queries get boosted

2. **Modified `cmd_recall()`** (line ~2056):
   - Changed from flat dedup+priority sort to per-query collection + RRF fusion
   - Each sub-query's results are kept as separate lists
   - `rrf_fuse(all_query_results, k=10)[:max_memories]` replaces the old sort

**Benchmark code changes (`quality/recall/run_benchmark.py`):**
- Added `expand_query()` — generates 3-8 sub-queries per benchmark query
- Added `rrf_fuse()` — RRF scoring implementation
- Added `--mode rrf|baseline` CLI flag (default: rrf)

**Round 2 metrics:**

|--------|---------|---------|-------|

**Key finding:** RRF dramatically reduced noise (+1.2) by rewarding consensus across sub-queries. Two remaining misses are hard NWS weather API queries.

### Round 3 Steps 2-3 — handler.js (COMPLETE)

Deployed two changes to `~/.openclaw/hooks/clawbot-memory/handler.js`:

1. **Multi-turn context:** Changed from single last user message to last 3 messages (any role) joined with `" | "`, truncated to 300 chars (was 200). Research basis: Mem0 reports +26% F1 on LOCOMO benchmark with multi-turn context.

2. **Trivial turn gate:** Added `TRIVIAL_PATTERNS` regex matching "ok", "thanks", "hello", etc. under 25 chars → early return, no recall. Research basis: A-MAC paper (ICLR 2026) — triviality is a tight semantic cluster, simple heuristic catches 95%+. Saves ~130ms + API cost per trivial turn.

### Round 4 Step 3 — Azure text_weights Verification (COMPLETE)

**Findings from Azure AI Search diagnostic:**

| Feature | Status |
|---------|--------|
| Semantic search | Free tier (1K queries/month) |
| Semantic config | `memory-semantic` on `content` field |
| Scoring profiles | **None existed** before Round 6 deploy |
| Vector search | HNSW (cosine, m=4, 3072-dim text-embedding-3-large) |

**Key insight:** text_weights ARE pre-reranking (L1) — they get overridden by semantic reranking (L2). However, since we're on free tier with limited semantic queries, text_weights still help for non-semantic queries. Kept in code.

### Round 7 — LLM Topic Expansion (COMPLETE)

**What was built:** `_llm_expand_query()` function + `_LLM_EXPANSION_CACHE` dict in `smart_extractor.py`.

**How it works:**
```
1. Try static domain map (23 topics)    → ~0ms, covers ~70% of queries
2. Check in-memory cache                → ~0ms
3. Call GPT-4.1-mini via Foundry proxy   → 0.4-2.6s, one-shot prompt
4. Cache result (per-process lifetime)
5. Fall back to word-based bigrams       → if LLM fails/times out
6. Dedup + cap at 8 queries
```

**Test results:**

| Query | Time | Example expansions |
|-------|------|--------------------|
| kubernetes pod scheduling | 2.6s (cold) | kubernetes scheduler, pod affinity, taints and tolerations |
| terraform state management | 1.5s | terraform state file, terraform state commands, remote state |
| webhook retry logic | 0.5s | webhook failure handling, delivery attempts |
| backup rotation policy | 0.4s | backup schedule, data retention policy, offsite backup |
| (cache hit) | 0.0s | (same as first call) |
| tailscale exit node (static) | 0.0s | VPN, wireguard, exit node, mesh network |

**Cost:** ~$0.001 per cache miss. <$0.01/month at current volume.
