# Decision: Contextual Re-indexing — Prefix Template for Memory Facts

**Date**: 2026-03-28
**Status**: Proposed
**Deciders**: Des Villa
**Research source**: plans/research-log.md — "Memory Retrieval Best Practices 2025-2026"

## Context

Anthropic's contextual retrieval technique (Sep 2024) prepends a 1-2 sentence context description to each document chunk before embedding. The result reduces failed retrievals by 49% in Anthropic's benchmarks, and by 67% when combined with contextual BM25. The technique is applicable to memory facts. Currently, memory facts in Azure AI Search are embedded as bare text (e.g., "The gateway uses claude-opus-4.6"). The tags, project, and date metadata exist in SQLite but are not embedded with the content.

## Options Considered

### Option A: No contextual prefix (status quo)
- **Pros**: No implementation cost; no re-indexing downtime
- **Cons**: Embedding of bare fact text misses metadata signals; query "what model does the gateway use" may not align well with "The gateway uses claude-opus-4.6" in embedding space if the query phrasing differs
- **Deployment impact**: None

### Option B: Metadata-only prefix (deterministic, no LLM)
- **Format**: `[{project} | {domain} | {type} | {date}] {content}`
- **Example**: `[openclaw_vm | infrastructure | type:architecture | 2026-02-20] The gateway uses claude-opus-4.6 as its primary model.`
- **Pros**: Zero LLM cost; deterministic; can be generated from existing SQLite fields; no hallucination risk
- **Cons**: Less natural language; may not help embedding alignment as much as a sentence-form prefix; BM25 benefits from keyword enrichment, which this provides
- **Deployment impact**: One-time re-sync via memory_bridge.py sync --full

### Option C: LLM-generated context sentence (Anthropic's original approach)
- **Format**: One sentence generated by GPT-4.1-mini: "This memory records that {paraphrase}, captured from project {project} on {date}."
- **Example**: "This memory records the primary LLM model configuration for the openclaw gateway, captured from the openclaw_vm project on 2026-02-20. The gateway uses claude-opus-4.6 as its primary model."
- **Pros**: Most natural language; best embedding alignment; follows Anthropic's tested approach exactly
- **Cons**: One LLM call per memory (~$0.003 total for 100 memories); introduces hallucination risk on rare cases; requires re-generation when content changes
- **Deployment impact**: LLM call cost + re-sync

### Option D: Hybrid (B + C combined)
- **Format**: Metadata tags as prefix + LLM sentence
- **Pros**: Both keyword enrichment (BM25) and semantic alignment (vector)
- **Cons**: Longer embeddings; more complex re-index pipeline
- **Deployment impact**: Same as C

## Decision

**Implement Option B (metadata-only prefix) initially. Upgrade to Option C on next quarterly re-index if contextual retrieval is validated on ClawBot store.**

Rationale:
- Option B provides immediate BM25 enrichment at zero cost — tags like "type:architecture" and "domain:infrastructure" appearing in the embedded text will boost keyword recall for tag-matched queries
- Option C's 49% improvement assumes LLM-generated context sentences. This can be tested in a staging index before committing to production
- The domain vocabulary in ClawBot memories is specialized enough that metadata tags already contain high-signal keywords

Prefix template for Option B:
```python
def build_contextual_content(memory: dict) -> str:
    tags = memory.get("tags", "")
    project = memory.get("project", "unknown")
    date = memory.get("created_at", "")[:10]
    content = memory["content"]
    prefix = f"[{project} | {tags} | {date}]"
    return f"{prefix} {content}"
```

This change goes into `memory_bridge.py` → `build_document()` before the embedding call.

## Consequences

**Easier:**
- BM25 keyword matching improves for tag-aligned queries
- Retrieval logs become more interpretable (the embedded text shows metadata context)

**Harder:**
- Re-indexing required on every significant tag change (rare)
- Staged approach (B → C) adds a future migration step
